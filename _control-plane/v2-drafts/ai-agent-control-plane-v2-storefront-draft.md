## Why This Exists

AI agents can generate output quickly. The hard part is deciding what is safe to accept, what needs human review, what evidence must exist before launch, and how product operations stay coherent once checkout, fulfillment, support, and updates enter the picture.

The v2.0 direction reframes the package as a commercialization upgrade for agent-assisted product work. The point is not only to manage implementation tasks. The point is to keep product, release, and buyer operations under explicit control.

## Best Fit

- Technical solo builders already using GitHub issues, pull requests, and agent workflows.
- Small technical teams that need agent output, human approval, and release readiness in one operating loop.
- Operators turning internal tooling or downloadable documentation packs into products with explicit commercialization steps.

## What You Can Control

- Turn product ideas into issue-shaped work with explicit evidence and next-decision fields.
- Require human acceptance before an agent task becomes product truth.
- Track package, page, checkout, fulfillment, support, and follow-up work in one control plane.
- Keep public-facing claims aligned with the real artifact and delivery path.

## Inside the Product

The future `#inside` section should be capability-grouped rather than presented only as a raw file inventory.

### Control Plane Core

- GitHub issue and project structure for shaping work, status, ownership, release stage, and proof requirements.
- Product lifecycle templates for turning a product idea into scoped execution, review, and launch work.
- Operating models that keep launch state, evidence, and next human decisions visible.

### Human Gates

- Review gate templates for acceptance, revision, rejection, and escalation.
- Launch-readiness and public-safety checklists that stop unsupported claims from going live.
- Approval packets that make the required human decision explicit before release or buyer-facing changes.

### Hermes/Telegram Operating Layer

- Supervision patterns for summaries, gate packets, and operator-visible status updates.
- Clear authority boundaries so automation can inform, route, and summarize without autonomously publishing, pricing, delivering, or messaging buyers.
- Notification guidance for keeping operating signals concise and reviewable.

### Commercialization Layer

- Offer and release readiness templates covering storefront, checkout, fulfillment, support, refund boundaries, and post-purchase operations.
- Commercialization workflows that keep product copy, artifact state, and delivery promises aligned.
- Buyer-fit and pre-purchase intake patterns for structured questions without exposing sensitive details.

### Knowledge/Evidence Layer

- Evidence collection templates for release checks, validation snapshots, change logs, and publish gates.
- Business knowledge and operating-reference material that ties decisions back to durable artifacts instead of chat-only status.
- Public-safe boundary guidance for what may and may not appear in storefront copy, public issues, and buyer-facing materials.

### CLI

- Project-local installation and verification tooling for supported agent workspaces.
- Commands intended to help buyers plan setup, initialize local scaffolding, and verify installation health.
- CLI claims should remain subordinate to the broader commercialization package and should not be presented as the whole value of v2.0.

## Boundaries

- Not a SaaS product.
- Not an autonomous publishing bot.
- Not an official integration from GitHub, Stripe, Telegram, or any third-party platform.
- Not custom consulting, custom setup, or ongoing managed support.
- Not a transfer of private operating history, customer data, secrets, or internal business records.

## Checkout and Delivery Draft Boundaries

Use this section only after the release checklist passes:

- Checkout must describe the actual active fulfillment path on publish day.
- If fulfillment is manual, say so plainly and keep the delivery window explicit.
- If fulfillment becomes automated, do not claim automation until a fulfillment drill has passed with the current artifact.
- Support scope must stay limited to package access, broken-file replacement, and clarification of included materials unless a separate offer says otherwise.
- Refund language must match the real post-purchase handling path and should avoid promising exceptions that operations cannot honor consistently.
- Future-version entitlement must only be claimed if the checkout or delivery note explicitly grants it.
